## Context

`permissions.ts` is a pi-coding-agent extension that intercepts `tool_call` events for the `bash` tool and prompts the user before executing sensitive commands. It currently covers two threat domains â€” destructive git operations (11 rules) and destructive GitHub CLI operations (7 rule groups + API method detection) â€” but has no protection against system-level dangerous commands. The official `permission-gate.ts` example covers the system domain (rm -rf, sudo, chmod 777) but lacks the session-approval UX, command-segment splitting, and shell-wrapper detection that make the current extension practical.

The extension lives at `dotfiles/pi/agent/extensions/permissions.ts` (~230 lines) and is linked into `~/.pi/agent/extensions/` by the dotfiles installer.

## Goals / Non-Goals

**Goals:**

1. Add `SENSITIVE_SYSTEM_PATTERNS` â€” a new rule array covering `rm -rf`/`--recursive`, `sudo`, `chmod`/`chown 777`, `dd`, `mkfs.*`/`fdisk`/`parted`, `shutdown`/`reboot`/`poweroff`/`halt`, and `curl`/`wget` piped to a shell (`bash`/`sh`/`zsh`/`fish`).
2. Add a privilege-stripping step inside `extractShellWrappedCommand` so `sudo bash -c "git push --force"` unwraps to the inner git command and yields a specific git reason, without breaking the generic sudo detection for un-nested commands.
3. Insert the system check as the last step in `checkSegment` after `git â†’ gh â†’ wrapped`.
4. Order system patterns from most specific (`rm -rf`) to most general (`sudo`) so the most informative reason wins.
5. Migrate `getApprovalScope` from `spawnSync` to `pi.exec`, passing `pi` as a parameter (like `dirty-repo-guard.ts`).
6. Replace JSON logging with the `web-tools.ts` pattern: `~/.local/state/pi/permissions.log`, auto-rotation at ~160 KB / 2000 lines, string-plain format.
7. Leave approval-dialog timeout behaviour unchanged (no timeout â€” user preference).

**Non-Goals:**

- No new npm dependencies or configuration files.
- No changes to the session-approval algorithm (`sessionApprovals` `Set` keyed by `scope + '\0' + normalizedCommand`).
- No changes to the command-segment-splitting logic (`splitCommandSegments`).
- No changes to the UI presentation of the three-option dialog ("Allow once" / "Allow for this session" / "Block").
- No support for `kill`, fork-bomb detection, `mv` to `/dev`, raw disk redirection, or expanded `chmod` patterns.

## Decisions

## Risks / Trade-offs

| Risk                                                                                                                                                                                     | Mitigation                                                                                                                                                         |
| ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `pi.exec` may not accept an explicit `cwd` option in its options object, making the scope-detection path dependent on the process's current directory.                                   | Verify the `pi.exec` signature from local package docs before implementation. Fallback: prefix command with `cd <cwd> &&` or keep `spawnSync` with explicit `cwd`. |
| The privilege-stripping regex `PRIVILEGE_RE = /^(?:sudo\|doas\|pkexec)\s+/i` may not cover all privilege-escalation commands across platforms (e.g., `runuser`, `su -c`).                | Start with the three most common (`sudo`, `doas`, `pkexec`). Extend only if users report false negatives.                                                          |
| `normalizeCommand` collapses quoted whitespace (`echo "a   b"` â†’ `echo "a b"`), producing incorrect approval keys for commands where internal whitespace is semantically significant.    | Accept as known edge case â€” git/gh commands rarely depend on multi-space arguments. Not in scope for this change.                                                  |
| Steamrolling `sudo` with system-pattern ordering (specificâ†’general) means `sudo chmod 777 file` will be caught as "chmod 777 opens permissions" rather than "sudo escalates privileges". | Block reason is still informative; the command is blocked either way. This is a strict improvement over the current state (no block at all).                       |
